# Lec11: Memory Management
## 物理内存
物理内存没有其他意义，只是存储（易失）运行信息的一个物理介质
‣ 连续字节/字（Byte）的数组，每行都有自己的地址（n 位地址总共有 $2^n$ 个字节）
‣ 通过地址可以写/读相应的字节
‣ 地址和数据分别通过地址总线和数据总线进行传播

## 保护
有了进程抽象，内存就有了语义
‣ 其代表了当前进程的“状态”
‣ 而这个“状态”应该遵循状态机（即进程）的规约而改变

![1780560961723](image/lec11MemoryManagement/1780560961723.png)
到底哪些"状态改变"操作是非法的(需要被保护拦截)？
![1780561086968](image/lec11MemoryManagement/1780561086968.png)
保护不是一开始就有的，在单道程序设计(Uniprogramming)年代，一次只能跑一个程序，应用程序总是装载到物理内存中的同一个位置，所以不需要保护机制。
多道程序设计(Multiprogramming)允许内存中有多个进程
既然内存里有多个进程，CPU 就可以在它们之间快速来回切换: 跑一会儿 A,保存 A 的状态(寄存器等),切去跑 B……
这个"保存/恢复状态再切换"的动作就是**上下文切换(**context switch)。
![1780561533717](image/lec11MemoryManagement/1780561533717.png)
但早期的多道程序设计没有保护机制，一个程序中的bug会导致其他程序也crash（包括 OS crash），程序也没有被禁止访问其他进程的地址

![1780561755672](image/lec11MemoryManagement/1780561755672.png)
一种通用的保护机制是**基于地址的保护机制**，用基址和界限来保护各自的内存
![1780561902731](image/lec11MemoryManagement/1780561902731.png)
![1780561955742](image/lec11MemoryManagement/1780561955742.png)
![1780562022830](image/lec11MemoryManagement/1780562022830.png)
![1780562039839](image/lec11MemoryManagement/1780562039839.png)
但是使用物理地址有缺点
物理地址对应用是可知的，导致：
- 一个应用会因其他应用的加载而受到影响(loader加载器压力骤增)
- 一个应用可通过自身的内存地址，猜测出其他应用的加载位置
是否可以让应用看不见物理地址？
- “看不见”，指应用对物理地址不可知
- 一个进程不用关心其他进程占了什么地址，不受其他进程的影响
- 看不见其他进程的信息，带来更强的隔离和保护能力

## 虚拟内存抽象
以虚拟内存抽象为核心的内存管理
- CPU:支持虚拟内存功能，新增了虚拟地址空间（通过Memory Management Unit, MMU单元）
- 操作系统: 配置并使能虚拟内存机制
- 所有软件: 均使用虚拟地址，无法直接访问物理地址

虚拟地址会被硬件自动转换成物理地址，应用程序不需要关心这个过程
每个应用程序有独立的虚拟地址空间
‣ 应用程序认为自己独占整个内存(透明性)
‣ 应用程序不再看到物理地址
‣ 应用程序加载时不用再为地址增加一个偏移量

### Linux虚拟地址空间
1. Process（进程）→ task_struct
每个进程在内核里都用一个 task_struct 来表示,它保存了进程的全部信息(PID、状态、寄存器等)。
2. Address space（地址空间)→ mm_struct
mm 指向 mm_struct,它描述整个进程的内存布局。图里列出的几个字段:
    - start_code:代码段起始地址
    - start_stack:栈的起始地址
    - start_brk:堆(heap)的起始地址
可以理解成一张"地图的总览",记录各个区域大致从哪里开始。
3. Virtual memory segment(虚拟内存区域)→ vm_area_struct
每一块连续且属性相同的内存区域(简称 VMA),用一个 vm_area_struct 描述:
    - vm_start / vm_end:这块区域的起止地址
    - vm_flags:权限标志(可读/可写/可执行)
    - vm_file:如果这块区域是映射某个文件,就指向那个文件
    - vm_pgoff:在文件中的偏移量
![1780562820919](image/lec11MemoryManagement/1780562820919.png)
一个进程的地址空间(从 0 到 0xffffffff)被划分成若干块。每一块(深蓝色)对应一个 VMA

这些 VMA 通过 next 指针串成一个链表(图中绿色梯形之间的箭头),每个 VMA 用 start、end 指向它在地址空间里覆盖的范围。整个链表由底部的 mm_struct 统一管理。

## 连续内存分配
最简单的内存分配方式，进程（包括操作系统内核）被分配一个**连续的物理内存地址**
利用基址和界限机制隔离用户进程之间的地址空间，以及防止用户进程修改操作系统的代码和数据
![1780573165284](image/lec11MemoryManagement/1780573165284.png)
多个进程时需要将物理内存进行分割，每个进程占据一个连续的物理内存分区，有两种分割方式
- 固定分区：物理内存一开始就被分为“固定”大小的区域，各个区域的大小可以相同，也可以不同，一个进程选择一个空闲区域进行分配
- 可变分区：每个分区大小不固定，分区数不固定

### 固定分区
![1780621931157](image/lec11MemoryManagement/1780621931157.png)
问题是：如果一个进程所需要的空间非常大，这种固定分区就不合适了
如果某个进程需要的内存超过所有分区中最大的那个，就无法放入该内存分区，进程无法运行
此外，进程的size各种各样，但只能选择一个分区存放，这个分区大于进程的部分就是一个内部碎片（internal fragmentation），其无法被其他进程占用，自己也不用
此外，固定的分区数也限制了可以同时放入内存的进程数

### 可变分区
![1780622146766](image/lec11MemoryManagement/1780622146766.png)
鉴于大多数进程在运行时会增长，在加载进程时会分配一些额外的内存。
此外，如果给进程所分配的区域用完，那么该进程将可能： 
- 被移动到一个有足够空间的空闲区域中 
- 被交换（Swap）出内存（到磁盘），直到能够创建一个足够大的空洞 
- 或者直接被终止（OOM Killer）

**碎片**：无法被分配的未使用内存
- 外部碎片（空洞）：由于分散的小的不连续的空闲空间导致的内存浪费，发生在分区之间，通常是由于进程的不断加载和释放造成
- 内部碎片：由于分区大小和加载的进程大小之间的差异（即进程小于分配的分区）导致的内存浪费，发生在分区内
![1780623075253](image/lec11MemoryManagement/1780623075253.png)
对于外部碎片，进程终止或暂时交换出磁盘，可以释放内存，可能会合并一些碎片形成大的可分配区域
通过**紧缩**（compaction）减少外部碎片化 
- 重排内存内容以将所有空闲内存放在一起 
- 时间复杂度较高（一般在系统实在没有内存的情况下才会做，内存分配slow path的一部分）
![1780623149706](image/lec11MemoryManagement/1780623149706.png)

#### 空闲内存管理
为了实现动态可变分区内存分配，操作系统应维护以下信息：
- 已分配的分区
- 空闲的分区（空洞）

一种简单的办法：Bitmap
- 内存被划分为分配单元（几字节到几千字节）
- 每个分配单元对应位图中的一位，如果该单元空闲则该位为0，如果被占用则该位为1（或反过来）
